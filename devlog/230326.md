# What was achieved/learned
- Toyed with different command buffers for different pipelines
	- Added another semaphore for this. so flow is work1 is submitted in the queue, with work2 semaphore as the signal, then work2 is submitted as well with the renderfinished as the signal and the semaphore for work1 as the wait.
- Implement the tutorial resize, through the vk::Result.
	- auto [result, imageIndex] = swapChain.acquireNextImage(UINT32_MAX, *presentCompleteSemaphore[frameIdx] /*Using this as a signal*/, nullptr); this returns a result
	  of outofdatekhr which allows us to call the recreateSwapChain function.
	- Same thing with the result of presentKHR, but in this case the frameBufferResized is also checked. this is to account for the times where drivers wont trigger the aforementioned outofdatekhr, hence we handle it explicitly with the help of glfw callbacks
- learned about deadlocks and why its VERY important that when you use signals(fences and or semaphores), you make sure that the work that submits and uses said signals GETS submitted if not
  DEADLOCK

## Pattern for frames in flight is as goes
<b>Frame N</b> <br>
fence starts initialized at signaled (applicable for frame 0)
<br>⬇<br>
wait for fence
<br>⬇<br>
acquire image, signal semaphore of work 1
<br>⬇<br>
check for out of date surface
<br>⬇<br>
reset fence
<br>⬇<br>
submit work 1, signal the rendered semaphore, and wait on semaphore of work 1
<br>⬇<br>
submit presentation, waiting for the rendered semaphore
<br>⬇<br>
check for out of date surface or explicitly check for the resized frame buffer via glfw
<br>⬇<br>
increment the frame index, use modulo to prevent the index from going over the specified frames in flight number


# Frictions
- Had a bug in which instead of resetting the image view vector, i accidentally resetted the image vector instead for the swapchain. Whilst this may have been a non-issue considering i was recreating this from everytime
  i recreated the swapchain, the way i created the image view was such that i emplaced back on the vector so everytime the resize happens, it pushes 3 more images. Since i didnt clear it,
  the image view passed into the rendering info was stale (since i wasnt clearing it) hence everytime i resized the window crashed, long story short be careful of the buffers im using.
- Also something to keep in mind was the order of destruction of this. While it may not matter now, the image view opened my eyes as its dependency goes as follows, swapchain images -> image -> image views.
  Another thing to be wary of
